# Pin 模块

## 概述

Pin 是 Lumina 的第四个核心模块，实现**跨项目依赖约束传递**。

与 Memory 存储决策记忆不同，Pin 聚焦于**点对点的定向推送**：当 Agent 在项目 A 的工作中发现某项约束、注意事项或规范需要项目 B 知晓时，可以通过 Pin 将这条约束精确推送到目标项目。目标项目通过 FIFO 队列按序消费，确保约束被有序接收和处理。

Pin 的核心价值在于让约束在**正确的项目之间流动**，而不是广播到所有项目或沉淀为全局记忆。

## 核心能力

- **点对点定向推送约束**：明确指定从哪个项目推送到哪个项目，精准传递依赖关系
- **FIFO 队列按序消费**：接收端按先进先出顺序处理约束，保证时序一致性
- **pending→consumed 不可逆生命周期**：约束被消费后进入 consumed 状态，不可回退，确保消费记录完整
- **alias_name 模糊匹配 + project_id 兜底项目标识**：支持通过别名数组灵活定位目标项目，同时以 project_id 作为精确兜底
- **超时提醒与 force 重读**：约束在队列中等待过久可触发超时提醒，支持 force 参数强制重新读取已消费记录

## Pin vs Memory 对比表

| 维度 | Pin | Memory |
|------|-----|--------|
| 核心概念 | 契约传递 | 决策记忆 |
| 数据方向 | 点对点定向（从项目 A 到项目 B） | 全局存储（供任意会话检索） |
| 消费模式 | FIFO 队列消费 | 条件检索 |
| 生命周期 | pending→consumed 不可逆 | active→deprecated→archived |
| 典型场景 | 通知目标项目注意事项 | 记录架构决策供后续遵循 |

## 存储策略

- **PostgreSQL**：Pin 记录完整数据、Project 实体数据、Info 表 pin_expire_days 配置

## 与其他模块的关系

Pin 模块完全独立，不与 RepoWiki、Memory 或 Q&A 直接交互。Agent 通过 MCP 自行决定何时推送和消费约束。

典型使用场景：
- Agent 在项目 A 工作时，发现某项规范变更会影响项目 B，调用 `pin_push` 将约束推送到项目 B
- Agent 进入项目 B 时，调用 `pin_get` 消费等待中的约束，按 FIFO 顺序依次处理
- 约束长时间未被消费时，系统根据 Info 表中的 `pin_expire_days` 配置触发超时提醒
- 需要回顾已消费的历史约束时，可通过 force 参数强制重新读取 consumed 记录

Info 表复用现有认证模块的 Info 表结构，新增 `pin_expire_days` 配置项用于控制约束的超时提醒周期。
